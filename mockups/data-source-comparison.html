TRENDING — DATA SOURCE

Two realistic ways to fetch creator data inside the desktop app. Both are technically scraping (no real API exists for tracking arbitrary creators). The difference is where the request comes from and how Instagram sees it.

A · OWN SCRAPER

A background process inside the Tauri app that hits Instagram's HTTP endpoints directly using requests / reqwest with no real browser involved.

1
Rust async task
Runs every 6h on a tokio interval (or on-demand from the UI)
↓
2
HTTP GET i.instagram.com/api/v1/users/{username}/info/
Spoofs Instagram's mobile app User-Agent + headers. No session token = limited to public data.
↓
3
Parse JSON response
Followers, post grid, view counts. Trial reels: NOT visible (server filters them out for non-tester accounts).
↓
4
Store in local SQLite
Compute deltas vs last fetch, push to UI via Tauri event.

PROS

  • Fast — single HTTP call per creator
  • No webview UI baggage
  • Runs in background even when app is in tray
  • Easy to schedule + batch

CONS

  • IP gets rate-limited fast — same IP hitting many creators triggers IG's bot detection
  • Endpoints change every few months → maintenance treadmill
  • Can't see trial reels at all (server-side filter)
  • If you log in to authenticate → IG can ban that account
  • ToS violation; risk of legal C&D for distribution at scale
// Rust — simplified own-scraper async fn fetch_creator(handle: &str) -> Result<Profile> { let url = format!("https://i.instagram.com/api/v1/users/web_profile_info/?username={handle}"); let res = client.get(&url) .header("User-Agent", "Instagram 269.0.0.18.75 Android") .header("X-IG-App-ID", "936619743392459") .send().await?; // 90% of the time: rate-limit error, captcha, or shadow-ban res.json().await }

B · APP VIA WEBVIEW
RECOMMENDED

A second hidden Tauri webview window that loads instagram.com with the user's actual logged-in session, then parses the rendered DOM via injected JS.

1
Hidden Tauri window opens instagram.com
First time: user logs in once. Cookies persist between launches in the webview's profile dir.
↓
2
Navigate to creator's profile URL
e.g. instagram.com/aurora.fields/ — IG renders the real page in our webview, exactly like a browser.
↓
3
Inject JS to read DOM + intercept GraphQL
Wait for posts to load, scrape stats from the rendered page. Tap into IG's own GraphQL responses by patching fetch().
↓
4
Trial reels handling
Webview is logged-in as YOUR account. If IG picked you as a tester, trial reels of THAT creator appear in your feed and we can scrape them. Otherwise: same blind spot as Option A.
↓
5
Send results back to main UI
JS calls window.__TAURI__.event.emit('creator-data', payload). Main window listens.

PROS

  • IG can't tell us apart from a real Chrome user — same cookies, same fingerprint, same JS execution
  • Rate-limit risk is whatever your normal browsing hits, not API-bot levels
  • Can see everything you can see manually — including trial reels IF you're an IG-chosen tester
  • Gets data on private accounts you follow
  • No reverse-engineering endpoints — IG's own JS does the fetching

CONS

  • Slower — needs to render full pages (~1-2s per creator)
  • Requires keeping the hidden webview alive (more memory)
  • Selector breakage when IG changes class names
  • Still ToS-violating; account ban risk if IG ever notices the JS injection
// JS — injected into the hidden IG webview const origFetch = window.fetch; window.fetch = async (...args) => { const res = await origFetch(...args); if (args[0]?.includes?.('/graphql/query')) { const clone = res.clone(); clone.json().then(data => { // Trial reels show up here when we ARE the tester audience window.__TAURI__.event.emit('ig-graphql', data); }); } return res; };

BOTTOM LINE

Webview is the way. It uses your real session, renders the real page, and is functionally indistinguishable from you scrolling Instagram in Chrome. Same fingerprint, same cookies, same rate limits.

Trial reels caveat — same for both: Instagram only shows trial reels to a curated set of non-followers IG picked as testers. If your account isn't in that audience for a given creator, the trial reel does not exist from your perspective. No API or workaround changes that. We can detect them with high reliability when they DO appear in your session, but we can't conjure them out of thin air.

Both routes violate IG's ToS. The webview path is much harder for IG to detect (it's just a logged-in browser). Risk of account ban is non-zero; risk of legal action against a self-hosted desktop tool is essentially zero in practice.

Recommend: Option B (webview), with a "Connect Instagram" first-run flow that opens the hidden webview to instagram.com, lets the user log in manually, then the cookie persists. We start the trending feature behind a settings toggle so it's opt-in.
